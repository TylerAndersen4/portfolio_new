01 Summary

I worked with a fictional client that wanted to move into the cloud storage and organization market. There were a few key features they knew they wanted, but I would need to back up these claims with data. It was up to me to refine this rough vision and begin developing a user-friendly cloud storage platform.


02 Problem

What features does the platform need to fit into the already crowded cloud storage and organization market?


03 Solution

For the final product, I designed a cloud storage platform that would allow users to create documents and spreadsheets, upload files, and share these files with friends and colleagues. The product is accessible on both desktop and mobile devices.

04 Competitive Analysis

Google Drive is easy to use and has many of the features the client was looking for (document creation, social sharing, file organization).

Pinterest showcases images nicely and is visual-first, but isn’t a productivity powerhouse like Google Drive.

Evernote has a modern visual design and is useful for organizing web links, but its UI can be overwhelming at times.

05 User Survey

In order to learn more about how people use cloud platforms, I used Slack, Reddit and my own Facebook network to gather participants for the user survey. The survey asked participants about their cloud storage habits, preferences, and frustrations.

08 User Stories

After creating the personas and using the data from the survey, I was able to create the user stories. I organized these by role, task, and value, and then sorted them according to their priority level. For the MVP I decided to focus on account creation, document creation, document organization, and file adding. This decision was based on user data that pointed to user’s most necessary features. 10 Sketches & Wireframes

I sketched out views for various states within the product: landing page, sign up, sign in, forgot password, add content, create document, upgrade account. Below you can see the sign up view and the dashboard view.

I tested the wireframes with three different users, and had a few important takeaways:

  • Would need to add a clear “Add” button in future iterations. Users were unclear on where to add new content.
  • The flow for organizing a document was unclear. I would need to make it clear where users can organize a document once they’re inside.
  • Users expect confirmation pages after upgrading their plans.
  • Confusing navigation on the left side of the dashboard.

11 Logo & Branding

I used mind mapping and word association to come up with different images and feelings for the brand name. I settled on IdeaDesk, which I thought captured both the creativity and productivity associated with the platform I was designing. With the name in mind, I went on to create various forms of the logo:

Concept

I wanted to give a sense of cluttered papers on a desk, since these are the rough forms of ideas. I started by sketching different stacks of papers.

Refinements

I took the sketches into Figma and started playing around with different arrangements and shadows. I thought the primary font was powerful, so I wanted to keep the logo fairly simple. I experimented with placing the text in different areas, and liked how the papers overlapped with “Idea.”

Final Logo

For the final logo I put I changed the colors of the papers to make the design more inviting and playful. The papers on the left represent chaotic ideas, as if they’re coming out of a box. The final design is simple yet bold.

The IdeaDesk color palette focuses on white and greys with bright pops of color. The palette brings a sense of fun and spontaneity to the brand.

12 Mockups & Testing

I conducted 2 remote and one in-person test the desktop and mobile designs. During this round of testing, I wanted to get feedback but also hone my user testing skills. I worked on letting users have freedom in the product instead of directing them with verbal cues too much. This allowed me to get more insightful feedback that would shape future iterations of the product in important ways.

While conducting user testing I received feedback that the original pricing view (left) was difficult to read. After a few iterations, I created the final pricing view (right). This view incorporates checkmarks in order to reduce reading and uses strategic color in order to guide the user through the information.

For the add file dropdown, I recevied feedback from users that this menu would be easier to navigate if there were icons. I learned an important lesson here that designers should aim to provide visual aids in order to help the user. This is also helpful for people whose don't speak English as their first language.

I used preference testing to decide between these two views, and 90% of users preferred the native iOS view on the right. I learned here that keeping native design reduces cognitive load for the user because they're seeing visual cues that they already recognize.

13 Final Product

IdeaDesk solves the client’s problem by providing a practical and intuitive cloud storage platform. I relied heavily on user data and research during the design phases in order to deliver a successful product.

The dashboard view is uses color coding to indicate file types. File previews are enlarged in order to showcase the content. On hover, users can see the collaborators within a file.

The document view uses clean lines and has an intuitive layout. Users can organize documents into folders and create new folder directly from the document view, without having to return to the dashboard.

The sign in page allows user to choose between social media or email sign in. The background is greyed in order to focus attention on the modal.

14 Conclusion

What are the next steps for IdeaDesk?

  • Build out the “social” view
  • Work on visual refinements
  • Build out a view for collaborating on a document
  • Create the profile settings page: remaining storage, avatar picture, etc.

What did I learn?

  • Always save iterations of work for case studies.
  • Sketches and wireframes should be thorough and neat. These need to be detailed in order to clearly communicate with stakeholders early on and conduct successful initial testing.
  • Build desktop and mobile prototypes at the same time.
